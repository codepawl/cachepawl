# vLLM Path C Observe/Advisory Milestone

Date: 2026-05-27

Cachepawl completed the vLLM Path C observe/advisory phase for
`Zyphra/Zamba2-2.7B-instruct` on vanilla `vllm==0.21.0`.

This milestone does not enable runtime substitution. It establishes a
non-invasive diagnostic workflow and records the remaining contracts required
before any future mutation cycle.

## What Is Available

`cachepawl diagnose-vllm` is the supported product surface from this phase. It
turns existing vLLM cache-plan observation artifacts into an advisory report
without requiring the main Cachepawl environment to import vLLM, use CUDA, use
NVML, modify vLLM, monkeypatch private paths, replace allocators, or return
Cachepawl plans to vLLM.

The advisory report is useful today because it identifies planner-level cache
overestimation for a hybrid workload and reports concrete memory metrics.

## Evidence Summary

- Planner-stage replay reached real vLLM planner inputs and called
  `vllm.v1.core.kv_cache_utils.get_kv_cache_configs`.
- Planner output matched the runtime scheduler cache config.
- `runtime_changed_during_replay=false`.
- Mutation-readiness structural checks passed, including planner schema,
  `num_blocks`, cache group count, cache tensor count, layer coverage, dtype and
  state dtype, block/page size, Mamba state shape, attention/Mamba group mapping,
  and estimated/useful byte consistency.
- Runtime observations resolved scheduler/KVCacheManager structure, block usage
  metadata, worker tensor layout, request-to-block assignment, attention
  block-table/view metadata, and attention metadata builders.

## Runtime Contract Findings

The live request observation resolved request-to-block assignment. A bounded
single-request run observed scheduler request id `0-aaa14650`, completion output
request id `0`, active block ids `[1, 2, 3, 4, 5, 6, 7]` after the first step,
and block-pool usage returning to its initial state after completion.

The Mamba/attention observation resolved the attention-side contracts. It
observed `21` block-table tensor metadata summaries and `7` attention groups.
Only tensor metadata was captured, not tensor payloads or large runtime objects.

## Why Substitution Is Not Enabled

Controlled substitution is not approved for this cycle. The remaining blockers
are Mamba-side contracts:

- `mamba_state_index_contract`: `mamba_state_idx` was reachable but empty for
  the live request.
- `mamba_state_tensor_contract`: no Mamba state tensors were safely reachable by
  stable runtime attributes.
- Runtime cache config reported `mamba_cache_mode: none`.

Approving mutation without those contracts would require relying on incomplete
runtime evidence or using disallowed techniques such as vLLM source edits,
monkeypatching, private allocator replacement, or unsupported scheduler/worker
layout changes.

## Next Step

The next product task is to package the observe/advisory workflow for users:

- Document how to collect vLLM observation artifacts in the durable vLLM
  environment.
- Document how to run `cachepawl diagnose-vllm` from artifact inputs.
- Keep examples explicitly advisory-only.
- Defer mutation work until Mamba state-index and state tensor contracts are
  observable, or vLLM exposes a supported integration seam.
