# In-depth Research Report: Optimizing the Hybrid Mamba-Transformer-MoE Architecture

The explosion of large language models (LLMs) between 2024 and 2026 has exposed serious physical and algorithmic limits of the traditional self-attention mechanism. The exponential growth of the Key-Value (KV) Cache when processing ultra-long context sequences has created a "memory wall" that cannot be overcome simply by scaling hardware capacity. In response, an ecosystem of hybrid architectures has emerged, combining the complementary strengths of the Transformer model, State Space Models (SSMs, exemplified by Mamba/Mamba-2), and Mixture of Experts (MoE) networks. Pioneering systems such as the NVIDIA Nemotron-3 Super line (which reaches 120 billion parameters but activates only 12 billion per iteration thanks to LatentMoE) and AI21 Jamba-1.5 (with a 1:7 interleave ratio of Attention to Mamba) are redefining inference performance benchmarks. However, this multi-layered combination gives rise to fundamental conflicts in memory management principles, low-level systems compute synchronization, and consistency in algorithmic routing. This report conducts a comprehensive in-depth data sweep, dissects the core technical bottlenecks, and proposes a breakthrough research roadmap to optimize inference for this group of hybrid models.

## Part 1: Literature Landscape

A survey of research and technical literature published from late 2024 through mid-2026 shows that the academic and industrial community is striving to solve two core problems of hybrid architectures: (1) Unified Memory Management to reconcile the conflict between KV Cache and State Cache, and (2) State-aware Algorithmic Routing for MoE to prevent information fragmentation. Below are in-depth analyses of the most foundational works, covering their core methods and the "research gaps" that remain unaddressed.

The first noteworthy research in memory management for hybrid architectures is the isolated design of the SGLang system, presented in technical documents about the Dual-Pool architecture. **The core method** of SGLang stems from recognizing the fundamental difference in memory access behavior between Transformer and Mamba. Attention layers produce KV Cache with a "fine-grained growth, short cycle" profile, in which each new token occupies only a few Kilobytes (KB) and is continuously allocated or reclaimed through the PagedAttention mechanism. By contrast, Mamba's State Cache exhibits a "large contiguous block, long retention" profile, requiring a static memory region on the Megabyte (MB) scale for each request stream and which must be kept intact until the sequence completes. To avoid VRAM fragmentation caused by the interleaving of these heterogeneous allocation units, SGLang uses a "Dual-Pool" design (two physically isolated pools). The Mamba pool (`HybridReqToTokenPool`) is managed on a per-request basis, while the Attention pool (`HybridLinearKVPool`) continues to be managed fine-grained via paging. Furthermore, SGLang extends the RadixTree structure into a MambaRadixTree to share SSM state across sequences with a common prefix. Although this resolves fragmentation, **the critical weakness** of this approach lies in its rigid resource allocation. Statically configuring the capacity ratio via the `--mamba-full-memory-ratio` parameter at service startup leaves the system unable to adapt dynamically. In diverse inference scenarios (dynamic workloads) where the long/short context ratio fluctuates continuously, this physical isolation inevitably leads to memory waste (virtual OOM), where one pool runs dry while the other still has free space.

The second research approach tackles memory management from a unified perspective: the Hybrid KV Cache Manager mechanism of vLLM. In contrast to SGLang, vLLM V1 attempts to bring both KV Cache and Mamba State into a single paged allocation system via `HybridKVCacheCoordinator`. **The core method** relies on a Page Alignment algorithm. For hybrid models (Case 4 in the vLLM architecture) where the hidden state size (`kv_hidden_size`) differs enormously between Attention and Mamba layers, vLLM forces the system to use a uniform page size. The system achieves this by inflating the block size of Attention layers until it matches the size of the Mamba-2 state. However, **the weakness** of this mechanism has produced a disaster in VRAM utilization. Experimental bug reports (e.g., Issue #37121 on the DGX Spark platform with the Qwen3.5-4B-AWQ model) show that vLLM's uniform treatment of these layers caused memory overestimation of up to 7×. Because Mamba (using GatedDeltaNet) has $O(1)$ memory complexity for its state but is allocated by the system according to a paging scheme designed for the $O(n)$ complexity of Attention, the actual utilization ratio is only 13.7%. The absence of an internal Asymmetric Paged Allocator is a research gap with extremely high practical value.

Turning to the algorithmic routing side, the third landmark study is the paper "Routing Mamba: Scaling State Space Models with Mixture-of-Experts Projection" published at NeurIPS 2025. **The core method** of Routing Mamba (RoM) addresses the performance degradation that occurs when MoE is naively integrated into SSMs. In standard Transformer-MoE designs, sparsity is applied only at the Feed-Forward (FFN) networks. However, the Mamba architecture has many complex projection layers (Conv Proj, Gate Proj, etc.). RoM turns the $W_{in}$ and $W_g$ projections into a set of sparse linear experts and, most importantly, shares routing decisions between these projection layers and the ultra-lightweight sub-modules inside the Mamba block. Through a routing network based on the softmax of the dot product $X_t \cdot W_r$, a 10B-parameter RoM model (activating 1.3B) can achieve performance equivalent to a dense Mamba model 2.3× larger. Even so, **the critical weakness** of this architecture is that the Router Network still computes weights purely from the input token embedding at time $t$ ($X_t$). The essence of Mamba's strength lies in compressing all sequence history into a hidden state ($h_t$). MoE routing that ignores $h_t$ means experts are selected based only on the local semantics of a single token, resulting in sequence-context loss when processing documents that are hundreds of thousands of tokens long.

The fourth study, closely related to context loss, is the paper "PaceLLM: Brain-Inspired Large Language Models for Long-Context Understanding" (NeurIPS 2025) and theories on State-aware MoE such as MoMamba. **The core method** of PaceLLM focuses on resolving "Semantic Fragmentation" — a major barrier that arises when FFN weights (or MoE experts) are not structured over time, causing long-range information to become fragmented and forgotten. PaceLLM proposes a Persistent Activity (PA) mechanism to track, reuse, and update FFN states in order to maintain semantic coherence across thousands of tokens. Similarly, MoMamba applies state-aware MoE to multi-task prediction problems. **The weakness** of these approaches lies in their systems-deployment cost. PaceLLM introduces very high external memory access latency because of the constant updates to its activation memory bank. If this "anti-semantic-fragmentation" theory were applied directly to the routing network of Mamba-MoE — i.e., turning the hidden state $h_t$ itself into a Sequence-level Routing signal — the field has yet to find a hardware-optimized architecture capable of running at high speed without creating communication bottlenecks.

Finally, the fifth study surveys the Quantization Mismatch problem, represented by papers such as "Q-Mamba: On Post-Training Quantization for Mamba Models" and the SAW-INT4 (System-Aware INT4 KV-Cache Quantization) research. **The core method** of SAW-INT4 demonstrates that Block-Diagonal Hadamard Rotation before INT4 quantization of the KV Cache nearly fully recovers accuracy without breaking the continuity of paged memory. At the other extreme, Q-Mamba and Quamba2 show that Mamba's hidden state is extremely sensitive to quantization because it contains outlier channels with very large values. To quantize Mamba down to W8A8 or W4A8, these studies must employ Decoupled Scale Quantization (DSQ) and Efficient Selectivity Reconstruction (ESR). **The serious weakness** appears when deploying hybrid models. When the system combines a KV Cache data stream (INT4 with Hadamard) and a State Cache (INT8/BF16 with DSQ), the fundamentally different numerical noise structures converge at the MoE routing layer. This mismatch skews the activation distribution and triggers routing collapse, and so far no unified kernel exists that can handle mixed-precision math across these multiple dimensions.

## Part 2: Bottleneck Analysis and Feasibility

Assume a project to build an entirely new Dynamic Allocator for the Hybrid Mamba-Transformer-MoE architecture, implemented in OpenAI Triton on top of the vLLM infrastructure. The design process encounters three system-level physical and algorithmic barriers: data locality, synchronization latency, and quantization incompatibility. The analysis below dives into these technical risks and proposes feasible mitigation directions.

### Data Locality and HBM Bandwidth

Simultaneously managing two completely opposite memory-access mechanisms on the GPU's High Bandwidth Memory (HBM) is a data-locality disaster. The Transformer's Attention layer requires scattered reads from small (KB-scale) page blocks dispersed throughout VRAM, due to the historical-token gathering nature of the KV Cache. Conversely, the state update in a Mamba layer requires a Parallel Prefix-scan across a huge contiguous memory block (MB-scale) and continuously overwrites the new state values.

When hybrid models (for example the Jamba structure with a 1:7 ratio of Attention to Mamba layers) execute prefill or decode, the GPU Scheduler is forced to switch repeatedly between these two memory-access patterns. The direct consequence is that the on-chip cache hierarchy is overwhelmed. The L1 and L2 caches of the Streaming Multiprocessor (SM) blocks are constantly thrashed by Mamba's massive data blocks, so that Attention's KV Cache tokens are evicted just after being loaded. As a result, HBM bandwidth becomes saturated by redundant load/store cycles before the Tensor Cores' compute throughput (FLOPS) can be fully utilized. To address this, an asymmetric PagedAttention algorithm is needed. By establishing a Virtual Page Table capable of clustering Mamba states into virtually contiguous VRAM regions while keeping the fine-grained KV Cache structure physically, the system can "fool" the GPU into improving the L2 cache hit rate.

### Synchronization in Multi-Parallel Environments

Deploying hybrid models at huge scale, such as Nemotron-3 Super (120B parameters), forces the use of a combination of Tensor Parallelism (TP) and Expert Parallelism (EP) across multi-GPU systems. This compute mesh creates a physical bottleneck at synchronization barriers.

The essence of Mamba is computing state evolution over time; therefore, the Mamba state $h_t$ must be finalized at the current layer before moving on to the next. However, when the data flow enters an MoE layer, the algorithm requires All-to-All global communication over NVLink or InfiniBand to dispatch tokens to the correct GPU holding the corresponding expert, and then combine them back. The conflict arises when Mamba's linear scan requires continuity, yet the system stalls waiting for the All-to-All network to finish shuffling tokens. Furthermore, load imbalance from dynamic routing — when a few "hot experts" receive too many tokens — increases the wait time at network nodes.

Newer technologies such as Cluster Launch Control (CLC) on the NVIDIA Blackwell architecture let SM clusters query hardware queues directly without using atomic semaphores on GMEM, reducing latency. However, to solve the problem thoroughly at the software level, the allocator and scheduler must implement Compute-Communication Overlapping, similar to the DeepSeek-V3 architecture, so that the Mamba state update of the second micro-batch executes while the first micro-batch is in flight through the All-to-All network.

### Quantization Mismatch

Quantization is indispensable for reducing VRAM pressure, especially for ultra-long-context models (such as Jamba 1.5 handling 256K tokens via ExpertsInt8). However, hybrid architectures face a Numerical Precision Mismatch problem when trying to optimize both cache types at the same time.

If the system decides to quantize Attention's KV Cache to INT4 (e.g., using Block-Diagonal Hadamard Rotation to preserve the PagedAttention structure) but is forced to keep Mamba's State Cache at higher resolution (BF16, W8A8, or W4A8 with Decoupled Scale Quantization) to preserve outlier channels, a discontinuity in feature space emerges. When information from these two streams converges at the MoE Router Network, the difference in numerical magnitude and in the structure of quantization noise warps the activation distribution. The router (especially LatentMoE architectures that compress the latent space before routing, as in Nemotron-3) is disturbed by INT4 features with narrow dynamic range, leading to "routing collapse" — i.e., the model continuously sends tokens to a small fixed set of experts instead of distributing them evenly, severely degrading convergence and overall inference quality (Perplexity/Accuracy).

| Technical Issue (Bottleneck) | Severity | Conflict Mechanism (Root Cause) | Feasible Solutions |
|---|---|---|---|

Keep separate physical data structures but unify the Virtual Mapping address table to maximize L2 cache hit rate. |

## Part 3: Open-source Ecosystem Survey

Developing a Proof of Concept (PoC) to address the bottlenecks above requires combining strong, highly modular open-source libraries. Below is a detailed survey of the available "building blocks" on GitHub, evaluated on suitability and customization ease for a systems engineer working solo.

First, for PagedAttention projects with integration potential for state-based models, the two most prominent ecosystems are vLLM and SGLang. The vLLM project (v1 Engine) already applies `HybridKVCacheCoordinator` for hybrid models, allocating KV Cache Groups that share physical tensors (`KVCacheTensor`). However, the code that handles state-size differences (in `vllm/v1/core/kv_cache_utils.py`) uses padding to force small Mamba layers into Attention's large page frame, causing massive memory waste. On the other side, SGLang (with SGLight and PD Disaggregation) fully separates the physical pool into `HybridReqToTokenPool` and `HybridLinearKVPool`. Intervening in vLLM is assessed as difficult because the memory-management core is deeply tangled with the CUDA backend and the scheduler. By contrast, SGLang has medium customization ease, because its clean physical isolation makes it more convenient to experiment with dynamic resizing algorithms without breaking the page pointers of another system.

Second, in the search for the most optimized current implementations of Flash-Decoding tailored to Mamba-2, the IBM Triton Backend project for vLLM stands out as a breakthrough solution. The IBM team successfully refactored vLLM's complex CUDA kernels onto a backend based entirely on the OpenAI Triton language. Notably, they developed a unified Mamba SSD kernel (`KernelChunkalignedMamba2`) that fully replaces the previous 5 separate kernels and thoroughly resolves memory-layout mismatches in Conv1D. Alongside it is the FlashInfer system, a C++/CUDA library providing a unified API for Attention, GEMM, and MoE, with strong support for hardware quantization formats such as FP8/FP4. Although FlashInfer offers extreme performance and deep support for MoE routing (DeepSeek-V3, Llama-4), its customization ease is only medium because of the C++ compilation barrier. By contrast, the IBM Triton Backend has very high customization ease. The Triton language lets a Python/AI researcher directly intervene in how the State Cache is overwritten asymmetrically without needing to deeply understand the GPU's instruction-set architecture.

Finally, for high-performance quantized KV Cache libraries, the ML-Rust/Boostr ecosystem and specialized libraries such as Quamba2 are setting new frontiers. Boostr provides a massive GGUF-compatible quantization format system (26 formats including Q4_0, TQ2_0) and supports built-in integration of Hybrid Mamba/Transformer architectures. Its MoE and LayerNorm layers are kernel-optimized across many hardware platforms. However, the Rust-language barrier and FFI (Foreign Function Interface) interop make Boostr's customization ease low for a PoC that needs fast iteration. A more viable option for customizing Mamba quantization is to use the open-source code accompanying the Quamba2 paper or the Q-Mamba system (PyTorch/Triton), combining block-reconstruction methods (ESR) to quantize Mamba state input down to W8A8/W4A8, then connect it into vLLM's Triton backend.

## Part 4: Research Plan Proposal

Given the serious shortage of Heterogeneous Memory Allocation systems and the open opportunity of non-traditional routing in MoE within hybrid architectures, this report proposes two breakthrough research directions. These directions directly exploit the "blind spot" of large AI labs — which typically tend to solve problems by Hardware Scaling or Extreme Quantization rather than restructuring the system flow at the foundational algorithmic level.

### Proposed Research Directions

**Direction 1 (System Level): Asymmetric Virtual Memory Paging (AVMP)**

This research aims directly at solving the memory-waste paradox in hybrid models. Instead of SGLang's rigid "Dual-pool" design with a fixed ratio, or vLLM's forced "Page Padding" mechanism that wastes 7× memory (Issue #37121), the project will introduce an AVMP memory manager. The technique uses a "Multi-resolution Page Table" that allows parallel mapping of Attention's small pages (KB-scale) and Mamba's contiguous blocks (MB-scale) into a single linear VRAM space. The algorithm optimizes external fragmentation without losing the continuity required for Mamba's state scan.

**Direction 2 (Algorithm Level): Latent State-Driven MoE Routing (LSDR)**

This research redefines the Router Mechanism inside the Mamba-MoE structure to solve the "Semantic Fragmentation" problem. Instead of using a softmax network operating on $X_t$ (the input Token Embedding at time $t$), the LSDR algorithm extracts and uses Mamba's hidden state $h_t$ directly — which has already accumulated the entire sequence context from the start — to compute expert-activation scores. By designing a combined routing function (e.g., $W_x X_t + W_h h_t$), the network learns to select experts based on the overall "sequence history" rather than the "current token's syntax" (Sequence-level Routing), directly enhancing long-range reasoning capability without introducing the external memory latency of the PaceLLM architecture.

### Experimental Roadmap

The combination of Direction 1 and Direction 2 will create a comprehensive research project that smoothly spans the system optimization layer up to the algorithm layer. The experimental roadmap below is structured into 3 stages (lasting approximately 4-5 months), providing sufficient evidence for publication at top conferences (such as NeurIPS, SysML, or OSDI).

#### Stage 1: Baseline Setup and Bottleneck Analysis (Benchmarking)

The goal of this stage is to precisely quantify latency, memory waste, and the Perplexity drop of current approaches, building a solid baseline for comparison.

- **Environment and Model Setup**: Set up baseline systems using the AI21 Jamba-1.5-Mini model (12B parameters, 256K context) and Qwen3.5-4B (using GatedDeltaNet and Attention). Deploy them on a test system with a configuration equivalent to NVIDIA DGX Spark or servers with H100 80GB GPUs.
- **Memory-System Measurement**: Enable the vLLM profiler and Nsight Systems to monitor memory allocation on the vLLM v1 branch. Extract the `total_cumulative_preemption_cnt` metric (measuring how often sequences are evicted from VRAM) and compute the VRAM Overestimation Ratio between allocated and actually used capacity.
- **Algorithmic Routing Survey**: Run a basic routing experiment on a traditional MoE architecture by analyzing the sparsity variance among experts on a long-document retrieval task. Collect heatmaps to demonstrate that routing based on $X_t$ leads to suboptimal expert selection and a lack of coherence at the tail of long sequences, producing semantic fragmentation.

#### Stage 2: Core Solution Development

This stage requires intense programming work to build Triton kernels for the multi-resolution allocator and the new routing mechanism.

- **System Engineering**: Intervene in the `HybridKVCacheCoordinator` class in the vLLM core. The engineer will rewrite the `unify_kv_cache_spec_page_size` algorithm to remove forced padding for Mamba layers. Then, leveraging the IBM Mamba-2 Flash-Decoding source code, build a custom OpenAI Triton kernel. This kernel will act as an allocation controller, allowing Static Launch Grids to steer asymmetric Paged Attention.
- **Algorithmic Modification**: Modify the Mamba-MoE block source in the Jamba configuration. Rebuild the MoE Router so the input gate transitions from pure $X_t$ to a linear function that also takes in $h_t$ (Latent State-Driven Routing). To prevent routing collapse caused by the Quantization Mismatch between KV Cache (INT4) and State Cache, the system will integrate an ultra-lightweight Decoupled Scale Quantization (DSQ) layer based on the Q-Mamba theory.

#### Stage 3: Comparative Evaluation and Documentation

The final goal is to prove the superiority of the new system through strict metric matrices covering both hardware performance and language quality.

- **Needle in a Haystack (NIAH) Recall Testing**: Run the NIAH test on maximum-length sequences (up to 256K tokens, matching Jamba-1.5's designed capacity). Compare the Recall Rate between the Baseline (Token-based Routing) and the proposed algorithm (LSDR). The expected data will show that LSDR, thanks to its resistance to "semantic fragmentation", delivers significantly more accurate Recall scores in deep context tiers.
- **Throughput and VRAM-Efficiency Testing**: Run continuous request streams (Continuous Batching) with randomly varying sequence lengths. Measure decoded Tokens per second (Tokens/s) and the actual VRAM Utilization Ratio. The AVMP system is expected to thoroughly resolve the 7× waste of vLLM V1's manager, unlocking VRAM space to raise the `max_num_seqs` parameter (total concurrent sequences served) by 2× to 3× on the same hardware configuration.
- **Mixed-Precision Impact Evaluation**: Monitor the inference-quality difference (Perplexity Shift) when Mamba DSQ state information (INT8) combined with KV Cache (INT4) passes through the MoE Router. The result will serve as solid evidence that the system handles mixed-precision workloads stably at scale.

By combining asymmetric Paged Memory infrastructure with a state-aware MoE routing mechanism, this proposal directly breaks through the most intractable technical bottlenecks of the existing hybrid-architecture LLM ecosystem. The approach — starting from the low-level systems core and extending up to the algorithmic topology — holds enormous potential for realizing multi-agent AI tasks with ultra-long context sequences while maintaining excellent inference performance under extremely constrained hardware budgets.
